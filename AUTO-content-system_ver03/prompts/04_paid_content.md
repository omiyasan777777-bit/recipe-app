# Phase 4: 有料部分作成

## 概要

商品設計書に基づき、有料コンテンツの本文を**たこすけ式構成法 + テンプレート駆動**で作成する。
各章がたこすけ式構成法の流れを内包し、読者の感情を動かしながらノウハウを伝える。

**目標**: 導入章 + ノウハウ章（全章）+ まとめ章を完成させる

**手法**: テンプレート駆動執筆 + たこすけ式構成法 + コンテンツ融合法 + 13要素下書き構造

---

## 入力情報

- `output/01_research.md`（競合リサーチレポート）
- `output/02_concept.md`（コンセプト設計書）
- `output/03_product_design.md`（商品設計書）

## たこすけ式構成法

全体構造と各章の両方でたこすけ式構成法の流れを意識する。

```
P = Problem（問題提起）: 読者の悩みを言語化し共感する
A = Amplify（問題の拡大）: 放置した場合の恐怖・損失を描く
S = Solution（解決策の提示）: 解決できることを示し安心させる
T = Testimony（実績・証拠）: 自分や他者の成功事例を示す
O = Offer（提案）: 具体的な行動を促す
```

**全体構造でのたこすけ式構成法:**
```
セールスレター（無料部分）: P → A → S → T → O（購入促進）
有料部分 導入章: P → S → 全体像提示
有料部分 各章: P（この章の課題）→ S（解決法）→ T（実例）→ 実践
有料部分 まとめ: 振り返り → O（行動喚起）
```

## 13ステップ下書き構造

各章の執筆前に以下の13要素を整理する。

```
① コンテンツテーマ: [この章で扱うテーマ]
② コンテンツの軸: [このテーマの核心・結論]
③ コンテンツタイトル: [章タイトル]
④ ○○とは？: [テーマの定義・説明]
⑤ ○○のメリット: [これをやる理由・実績]
⑥ 結論: [軸の結論を明示]
⑦ 結論の説明/実行ステップ: [具体的な手順]
⑧ まとめ＋行動促進: [章のまとめと次のアクション]
⑨ P: 問題提起: [読者の壁を提示]
⑩ A: 問題の拡大: [放置した場合の悪夢]
⑪ S: 解決への安心: [解決できることを示す]
⑫ T: 実績者の存在: [自分以外の成功例]
⑬ O: ユーザーへの提案: [この章を学ぶべき理由]
```

**本文の構成順序:**
- テキスト版: ⑨→⑩→⑪→⑫→⑬→①→④→⑤→⑥→⑦→⑧

## コンテンツ融合法

競合3商品の良い要素を融合し、自分の独自要素を加えてオリジナルを作る。

```
■ マージの手順
1. 競合A: [取り入れる要素] ← 何が優れているか
2. 競合B: [取り入れる要素] ← 何が優れているか
3. 競合C: [取り入れる要素] ← 何が優れているか
4. 自分の独自要素: [何を加えるか]
5. 融合 → リライト → オリジナル化

■ マージのルール
- 構成は参考にするが、文章はすべて自分の言葉で書く
- 各商品から取り入れるのは「構造」と「切り口」であり「表現」ではない
- 必ず自分の体験・実例を加えてオリジナリティを出す
- 3商品の弱みを補完する形で独自コンテンツを追加する
```

## PART A: 第1章【導入】の作成

### 導入章の構成（13ステップ⑨→⑧の順序）

```markdown

{⑨ P: 読者の悩みを箇条書きで3〜5個}

{⑩ A: このまま解決しないとどうなるか2〜3文}

{⑪ S: 「でもご安心ください」+ 解決策の提示}

{⑫ T: 自分の実績 or 他者の成功例を1〜2つ}

{⑬ O: 「この教材で解決します」の提案}

{④ コンセプト1行要約}

{⑤ なぜこの手段でうまくいくのか理由を2〜3文}

---

## 全体の流れ

{⑥ 大ステップを番号付きで列挙}
{各ステップの1行説明}

---

{⑦ 各章のタイトルと概要を2〜3文ずつ}

{⑧ 読んだ結果の理想の未来 + 行動促進}

それでは2章で{大ステップ1の名称}について語っていきます！
```

## PART B: 第2章以降【ノウハウ章】の作成

### ノウハウ章 = 2パーツ構成

```
ノウハウ章
├── PART 1: 章の導入文（たこすけ式ミニ構成）
│   ├─ P: この章で扱う課題
│   ├─ A: 放置した場合の問題
│   ├─ S: この章で解決できること
│   ├─ 全体像（この章のステップ一覧）
│   └─ テーマの重要性（なぜやるのか）
│
└── PART 2: ノウハウ解説部分（小ステップの数だけ繰り返す）
    ├─ 課題の提示（読者の壁）
    ├─ 解決策の全体像
    ├─ 具体的な手順（ステップバイステップ）
    ├─ 実例・体験談
    └─ ワーク / テンプレート / チェックリスト
```

### PART 1: 章の導入文テンプレート

```markdown
# 第{N}章：{大ステップ名}

{P: この章のテーマに関する読者の課題を1〜2文}

{A: この課題を放置するとどうなるかを1〜2文}

でもこの章を読めば...

{S: この章で何が解決するかを2〜3文}

具体的にはこんなステップで進めます。

{小ステップを箇条書きで列挙（各1行の説明付き）}

{このテーマをやることでどんなメリットがあるか2〜3文}

それでは具体的に見ていきましょう！
```

### PART 2: ノウハウ解説テンプレート

```markdown
## {小ステップ名}

### この段階での課題

{読者がここで直面する壁を1〜2文}

### 解決策

{小ステップ名}では、
**{何をするか1文で}** を解説します。

{なぜこれをやるのか理由を1〜2文}

これができると...

{メリットを箇条書きで3つ}

### 具体的なやり方

やることとしては、
**{作業の要約を1文で}** です。

**具体的な作業ステップ:**

{ステップを番号付きで列挙}

一つずつ解説していきます！

---

**STEP 1: {ステップ名}**

{具体的なノウハウを解説}
{数字・分量を明確に}
{自分の体験談・実例を含める}

---

**STEP 2: {ステップ名}**

{具体的なノウハウを解説}

---

（以下、小ステップ内のSTEP数だけ繰り返す）

### ✅ POINT

{補足情報・よくある質問・注意点}

### 📝 ワーク

{読者が手を動かす演習}
```

### ノウハウ部分の執筆ルール

| ルール | 詳細 |
|--------|------|
| 具体性 | 数字・分量を必ず明記する（❌「いくつか」→ ⭕「3つ」） |
| 段落制限 | 1段落は最大4行まで（スマホ読み対応） |
| 太字活用 | 重要キーワード・結論は `**太字**` にする |
| 箇条書き | 3項目以上の並列情報は箇条書きにする |
| 画像指示 | ノウハウの可視化が必要な箇所に `[IMAGE: 説明]` を入れる |
| ワーク | 読者に手を動かさせる「ワーク」を各章に1つ以上入れる |
| POINT | 補足情報は `✅POINT` として囲み要素にする |
| 体験談 | 各章に最低1つ、自分の実体験エピソードを入れる |
| 橋渡し | 各章の最後に次章への接続文を入れる |
| たこすけ式構成法 | 各章の冒頭でP→A→Sのミニ構成を入れる |

## 章ごとの文字数基準【厳守】

> **⛔ 文字数は必ず守ること。**
> 各章の執筆後に文字数をカウントし、基準を下回っていたら加筆してから次の章に進む。
> 合計文字数が基準範囲内に収まっていることを最終確認する。

**商品タイプは自己分析レポート（00a_self_analysis.md）で選択済み。**
選択されていない場合は、商品設計書（03_product_design.md）の価格から自動判定する。

### フロントエンド商品（15,000〜20,000文字）

| 章タイプ | 最低文字数 |
|----------|-----------|
| 導入章 | 2,000字以上 |
| ノウハウ章（小） | 3,000〜4,000字以上 |
| ノウハウ章（大） | 4,000〜5,000字以上 |
| まとめ章 | 1,500字以上 |
| **合計** | **15,000〜20,000字** |

### バックエンド商品（30,000〜40,000文字）

| 章タイプ | 最低文字数 |
|----------|-----------|
| 導入章 | 3,000字以上 |
| ノウハウ章（小） | 5,000〜6,000字以上 |
| ノウハウ章（大） | 7,000〜10,000字以上 |
| まとめ章 | 2,000字以上 |
| **合計** | **30,000〜40,000字** |

**バックエンド商品で文字数を稼ぐ手法:**
- 各ステップの解説がより詳細（手順を省略しない）
- ケーススタディ・事例が各章2〜3個
- ワーク・テンプレートが充実（穴埋め式・記入例つき）
- FAQ・トラブルシューティングが各章に付属
- 上級者向けの応用テクニック・補足セクションが追加

### 文字数の確認ルール【必須】

**各章の執筆完了時に、以下を必ず実行する:**
1. その章の文字数をカウントする
2. 基準を下回っていたら、以下の手法で加筆する:
   - 具体的なエピソード・体験談を追加する
   - 実例・ケーススタディを増やす
   - ステップをより詳細に分解して解説する
   - 「よくある質問」「よくある失敗」セクションを追加する
   - ワーク・演習の手順を丁寧に書く
3. 基準を満たしたことを確認してから次の章へ進む

**全章完成後に、合計文字数を必ず表示する。**
- フロントエンド: 15,000〜20,000字の範囲内か確認
- バックエンド: 30,000〜40,000字の範囲内か確認

## 全体の執筆プロセス【章ごとファイル分割 + 実測カウント方式】

**重要: 必ず1章ずつ分割して執筆すること。全章を一度に書かない。**

**一時保存先:** `output/chapters/` フォルダに章ごとのファイルを保存する。

```
STEP 1: 準備
  - `mkdir -p output/chapters` でフォルダを作成
  - 各章の13ステップ下書きを整理する
  - 商品タイプ（フロントエンド/バックエンド）と各章の目標文字数を確認する

STEP 2: 第1章（導入）を執筆
  - 執筆した内容を `output/chapters/ch01.md` に保存
  - bashで文字数を実測: `wc -m output/chapters/ch01.md`
  - 最低文字数を下回っていたら加筆して再保存 → 再カウント
  - 基準を満たしたら次へ

STEP 3: 第2章を執筆
  - 執筆した内容を `output/chapters/ch02.md` に保存
  - bashで文字数を実測: `wc -m output/chapters/ch02.md`
  - 最低文字数を下回っていたら加筆して再保存 → 再カウント
  - 基準を満たしたら次へ

STEP 4: 第3章〜最後のノウハウ章まで、1章ずつ繰り返し
  - 各章を `output/chapters/ch03.md`, `ch04.md`... に保存
  - 毎回 `wc -m` で実測し、基準を満たすまで加筆する

STEP 5: 最終章（まとめ）を執筆
  - `output/chapters/ch_final.md` に保存
  - `wc -m` で実測

STEP 6: 全章を結合
  - bashで結合: `cat output/chapters/ch*.md > output/04_paid_content.md`
  - 合計文字数を実測: `wc -m output/04_paid_content.md`
  - フロントエンド: 15,000〜20,000字 / バックエンド: 30,000〜40,000字 の範囲内か確認
  - 範囲外なら調整（不足→加筆、超過→不要な冗長部分を削除）

STEP 7: 結合後の調整
  - 章間の接続（橋渡し文）
  - 文体の統一（世界観設計のワーディングに合わせる）
  - 改行・装飾の調整
  - [IMAGE: ] 指示の確認
  - たこすけ式構成法の一貫性確認
  - 最終文字数を `wc -m` で再確認

STEP 8: コンテンツ融合法の最終確認
  - 競合3商品の良い要素が取り入れられているか
  - 自分の独自要素が十分に含まれているか
  - 全体として独自のオリジナル作品になっているか

STEP 9: 一時ファイルの削除
  - `rm -rf output/chapters/`
```

**⛔ 各章の執筆フローまとめ（厳守）:**
```
書く → output/chapters/chXX.md に保存 → wc -m で実測
→ 基準未満なら加筆 → 再保存 → 再カウント → 基準達成まで繰り返す
→ 基準達成したら次の章へ
```

**1章ずつ書く理由:**
- 一度に全章を書くと各章のボリュームが薄くなる
- 1章に集中することで具体性・深みのあるコンテンツになる
- `wc -m` による実測で文字数のごまかしが不可能になる
